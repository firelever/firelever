# Glossary — every term in this project, explained

Each entry: what it means, then how this repo uses it. (Interview tip: the second
half of each entry is your answer's second sentence.)

## Models

**LLM (large language model)** — A neural network trained on huge text corpora to
predict the next token, which turns out to generalize into following instructions,
writing, and reasoning. *Here:* Claude Opus 4.8 powers the pipeline agents, Q&A, and
triage; a fine-tuned Qwen handles classification.

**Parameters / "7B"** — The learned weights of the network; count is the rough
capacity measure. A 7-billion-parameter model fits on a laptop; frontier models are
orders of magnitude larger. *Here:* Qwen2.5-7B-Instruct is our small model.

**Frontier model vs. open-weights model** — Frontier: the strongest commercial models
(Claude, GPT, Gemini), accessed via API, weights private. Open-weights: models whose
weights you can download and run (Qwen, Llama, Mistral). *Here:* the slice 6 benchmark
is exactly this trade-off, measured.

**Qwen** — Alibaba's open-weights model family; Qwen2.5-7B-Instruct is a strong small
instruction-following model with a permissive license. *Here:* the student model.

**Instruct model / chat template** — Base models just continue text; "-Instruct"
variants are post-trained to follow conversations. The chat template is the exact
text format (special tokens marking system/user/assistant turns) the model was
trained on — use the wrong template and quality collapses. *Here:* train.py applies
Qwen's template via Unsloth.

**Token / tokenizer** — Models read text as subword units (~4 chars each in English).
Pricing, context limits, and latency are all measured in tokens. *Here:* the QA log
records token usage per answer to track cost per query (<$0.05 target).

**Context window** — The maximum tokens a model can attend to per request. *Here:*
retrieval exists partly because stuffing whole corpora into context is costly and
degrades attention; we retrieve 6 chunks instead.

## Fine-tuning

**Fine-tuning** — Continuing training on a pre-trained model with your own examples so
behavior moves toward your task. Contrast with prompting (no weight changes). *Here:*
slice 6 fine-tunes Qwen on 270 labeled triage examples.

**SFT (supervised fine-tuning)** — Fine-tuning on input→output pairs, teaching the
model to imitate the outputs. *Here:* email in, JSON classification out.

**LoRA (low-rank adaptation)** — Instead of updating all 7B weights, freeze them and
train small low-rank matrices inserted alongside key layers — under 1% of the
parameters, so training is fast and cheap and the "adapter" is tiny. `r` (rank) sets
adapter capacity. *Here:* rank 16 across attention and MLP layers.

**QLoRA / 4-bit loading** — LoRA on top of a base model whose frozen weights are
compressed to 4-bit precision, cutting VRAM ~4x so a 7B trains on a 24GB card.
*Here:* `load_in_4bit=True` in train.py; why a $0.16/hr GPU suffices.

**Quantization** — Storing weights at lower numeric precision (4-bit instead of
16-bit) to shrink memory and speed inference, at a small quality cost. *Here:* the
exported model is q4_k_m (a llama.cpp 4-bit scheme).

**GGUF** — The file format used by llama.cpp for quantized models; the standard for
running LLMs locally. *Here:* train.py exports GGUF so the model runs on a Mac.

**llama.cpp / Ollama** — llama.cpp: the C++ engine that runs GGUF models on ordinary
hardware. Ollama: a friendly wrapper around it (model registry, HTTP API). *Here:*
the benchmark serves the student via Ollama on Apple Silicon.

**Distillation (teacher-student)** — Training a small model to reproduce a large
model's outputs, trading peak quality for cost/speed/privacy. *Here:* Opus labels
synthetic emails (teacher); Qwen learns to mimic (student); ADR-006.

**Synthetic data** — Training examples generated by a model rather than collected.
Fast and cheap; risk is distribution mismatch with reality. *Here:* 300 generated
emails bootstrap training while real labeled traffic accumulates.

**Contamination** — When evaluation data leaks into training data, inflating scores.
*Here:* the eval set is hand-written and never touched by the corpus generator;
generation used disjoint scenario seeds. Say this proactively in interviews.

**Epoch / learning rate / train-val split** — One pass over the training data; the
step size of weight updates; and holding out data (our 30 emails) to detect
**overfitting** — memorizing training data instead of generalizing.

**Unsloth** — A training library that makes LoRA fine-tuning ~2x faster and much more
memory-efficient via hand-optimized kernels. *Here:* train.py is built on it.

**VRAM** — GPU memory; the binding constraint for model work. 7B in 4-bit ≈ 5GB plus
activations and optimizer state. *Here:* why the 24GB A5000 tier is enough.

## RAG (retrieval-augmented generation)

**RAG** — Retrieve relevant passages from your documents, put them in the prompt, and
have the model answer from them — grounding generation in private/current data
without retraining. *Here:* the entire copilot Q&A path (slices 2-3).

**Embedding** — A vector (list of numbers) representing text meaning; similar
meanings land near each other, enabling semantic search beyond keyword match.
*Here:* bge-small-en-v1.5 runs locally, 384 dimensions per chunk.

**Vector search / cosine similarity** — Finding stored embeddings nearest to the
query embedding; cosine measures the angle between vectors. *Here:* sqlite-vec does
KNN inside SQLite.

**Chunking** — Splitting documents into retrieval-sized pieces. Bad chunking is the
#1 silent RAG killer. *Here:* heading-aware ~700-char chunks; our eval caught a
splitter bug that broke "Instantly.ai" on the period.

**BM25 / keyword search** — Classic lexical ranking (term frequency, rarity, length
normalization); exact-match strength complements embeddings' fuzziness. *Here:*
SQLite FTS5 provides it.

**Hybrid retrieval** — Running keyword and vector search together and fusing results;
each catches what the other misses. *Here:* hybrid beat both alone, 86.4% vs 79.5%.

**RRF (reciprocal rank fusion)** — Merging ranked lists by scoring each item
1/(k+rank) summed across lists — no score calibration needed. *Here:* we tuned k=5
against the eval set after the textbook k=60 drowned single-list winners.

**Reranker** — A second, more accurate model that re-scores the top retrieved
candidates. Quality boost at latency cost. *Here:* deliberately deferred; on the
roadmap for the known misses.

**recall@k / MRR** — Retrieval metrics: did a relevant chunk appear in the top k; and
how high did the first relevant one rank (mean reciprocal rank). *Here:* the
retrieval eval's two numbers.

**Grounding / faithfulness / citation** — Grounded = every claim traceable to a
source; faithfulness = the measured % of answers fully supported; citations = the
[n] markers pointing at sources. *Here:* 100% measured faithfulness, chunk-level
citations.

**Refusal** — Declining to answer when sources don't contain the answer, instead of
guessing. *Here:* an explicit `answerable=false` in structured output; 100% refusal
accuracy on trap questions.

## Evaluation

**Eval / golden set** — A fixed, labeled test set with metrics, run repeatedly —
"unit tests for model behavior." *Here:* four of them (retrieval, QA, triage,
history-tracked), wired to fail on regression.

**Baseline** — The dumb-but-honest comparison your fancy approach must beat. *Here:*
keyword search for retrieval (79.5%), regex rules for triage (83.3%), Opus for the
student model.

**LLM-as-judge** — Using a strong model to grade outputs at scale (with human spot
checks, since judges err). *Here:* the QA faithfulness judge.

**p95 latency** — The time under which 95% of requests complete; tail latency matters
more than average. *Here:* PRD target of answer start < 5s p95.

## Safety & architecture

**Prompt injection** — Adversarial instructions hidden in data the model reads
(documents, emails) trying to hijack behavior. *Here:* every surface treats content
as data-not-instructions; a booby-trapped eval email is correctly filed as spam.

**Guardrails / human-in-the-loop** — Hard constraints outside the model. *Here:*
nothing ever sends without human approval; approval lives in a CLI/UI, deliberately
not exposed to any model-driven surface.

**Tenant isolation (multi-tenancy)** — One deployment, many customers, provably
separated data. *Here:* tenant_id on every row, vector-search partition keys, API
keys scoped per tenant.

**MCP (Model Context Protocol)** — An open protocol where servers expose
tools/resources with self-describing schemas that any AI client discovers at runtime
— "USB-C for AI integrations." *Here:* slice 1's read-only server over the lead
pipeline; slice 5b adds an HTTP variant.

**Structured outputs** — Constraining a model to return schema-valid JSON instead of
prose, making outputs machine-checkable. *Here:* every classification and the QA
answer object; refusal is a typed boolean, not parsed text.

**stdio vs. HTTP transport** — How an MCP client reaches a server: spawned locally
over stdin/stdout (no network, no auth surface) vs. hosted over HTTP (needs auth).
*Here:* ADR-001 chose stdio first, deliberately.

**Serverless GPU** — GPU compute billed per-second with scale-to-zero, vs. renting an
always-on machine. *Here:* RunPod pod for training; serverless is the plan if the
student ever serves production.

**ADR (architecture decision record)** — A one-page document per significant choice:
options, trade-offs, decision. *Here:* six of them — likely the most
interview-valuable artifacts in the repo.
